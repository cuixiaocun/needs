## 一、产品概述

### 1.1 产品定位

> 用 AI 高效收集农产品供需信息，智能撮合买卖双方与运输方，减少中间环节，降低流通成本。

### 1.2 核心价值主张

- **效率提升**：AI 对话替代传统人工沟通
- **成本降低**：减少中间环节，降低流通成本
- **智能匹配**：多维度自动撮合
- **服务托底**：代理人处理 AI 无法解决的复杂场景
- **明码标价**:服务费透明，公平合理

### 1.3 业务范围

- **第一阶段**：山东省试点（自营 + 第一个租户）
- **第二阶段**：通过 SaaS 模式复制到其他地区
- **商品范围**：蔬菜、水果等农产品，可销售至全国

## 二、用户角色

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| **卖方** | 果农、菜农、合作社 | 快速找到买家，卖个好价钱 |
| **买方** | 超市、批发商、餐饮企业 | 稳定供货，质量可靠，价格透明 |

| **管理员** | 平台运营人员 | 监控数据、审核用户、管理内容 |

### 3.2 模块一：AI 聊天收集需求

#### 3.2.1 技术方案
- **接入第三方大模型 API**（推荐：DeepSeek / 通义千问 / 文心一言）
- 自定义 Prompt 模板，控制对话范围
- 简洁无废话，像填表一样高效

#### 3.2.2 收集字段

| 角色 | 收集字段 |
|------|---------|
| **卖方** | 品类、数量、等级、产地、可供日期、期望价格 |
| **买方** | 品类、数量、质量要求、到货时间、目标价格、配送地址 |
| **运输方** | 车型、载重、区域、可出发时间、报价 |

#### 3.2.3 注册流程（重要更新）

```
用户进入 App
    ↓
直接和 AI 对话（无需注册，零门槛体验）
    ↓
AI 收集完整需求信息
    ↓
用户准备退出 / 关闭对话
    ↓
弹窗：「您的对话内容很有价值，是否保留？
       保留需要绑定手机号，下次进入可继续」
    ↓
  ┌─选「保留」→ 手机号注册 → 内容保存到账户 → 发布需求
  └─选「不保留」→ 直接退出，对话清空
```

#### 3.2.4 关键设计要点
- 未注册状态下的对话内容暂存在本地（或临时 Session）
- 对话内容自动生成结构化「需求卡片」
- 用户确认后才能正式发布到需求广场

### 3.3 模块二：智能撮合

#### 3.3.1 撮合维度
- 品类匹配
- 数量匹配
- 地区匹配
- 时间匹配
- 价格区间匹配

#### 3.3.2 撮合流程

```
需求入库
    ↓
多维度计算（算法打分）
    ↓
找到匹配度最高的对方
    ↓
双方接收通知
    ↓
双方选择「接受」或「拒绝」
    ↓
都接受 → 进入「待支付撮合费」状态
    ↓
任一方拒绝 → 返回需求池继续匹配
```

### 3.5 模块四：代理人系统 + 提成机制 ⭐ 重点模块

#### 3.5.1 代理人定位
- 平台自有团队（不招募第三方）
- 第一期 2 人，后期按业务量增加
- **靠提成激励，无需排班**
- 处理 AI 无法解决的复杂场景

#### 3.5.2 触发条件
- AI 三轮对话仍无法明确需求
- 用户主动点击「转人工」
- 撮合后双方对价格有分歧
- 复杂订单（比如多方拼单、特殊运输要求）

#### 3.5.3 订单分配机制

```
触发人工介入
    ↓
系统自动分配（轮询模式）
    ↓
按顺序分配给当前任务最少的代理人
    ↓
代理人后台收到通知
    ↓
代理人接单 → 处理 → 撮合成功
    ↓
代理人获得对应提成
```

#### 3.5.4 代理人提成机制（重点）

**提成规则**：
- 提成比例：**固定 20%**（基于平台收取的撮合服务费）
- 计算示例：
  ```
  订单金额：10000 元
  撮合服务费：500 元（5%）
  代理人提成：500 × 20% = 100 元
  平台净收入：500 - 100 = 400 元
  ```

#### 3.5.5 代理人 Web 后台功能

| 功能模块 | 包含内容 |
|---------|---------|
| **任务工作台** | 待处理任务、进行中任务、已完成任务 |
| **客户沟通** | 内置 IM 工具，与买卖双方实时沟通 |
| **订单管理** | 我服务的订单列表、订单详情、订单状态 |
| **提成统计** | 我的总提成、本月提成、提成明细 |
| **提现申请** | 提现金额、提现记录、审核状态 |
| **个人中心** | 个人信息、密码修改、登录日志 |

### 3.7 模块六：撮合服务费收取 ⭐ 重点模块

#### 3.7.1 收费规则

| 项目 | 规则 |
|------|------|
| **收费比例** | 订单金额的 **5%** |
| **付费方** | 买卖双方各付 50%（即各付 2.5%） |
| **收费时机** | **撮合成功时立即收取**（订单前收费） |
| **明码标价** | 需求广场每条需求都显示服务费 |
| **支付方式** | 支付宝 / 微信支付 |
| **资金流向** | 直接进入平台账户（山东自营版本进入你的账户） |

#### 3.7.2 收费流程图

```
AI 撮合成功
    ↓
通知买方和卖方
    ↓
双方都点击「接受撮合」
    ↓
系统生成订单
    ↓
显示：「请支付撮合服务费 XX 元」
    ↓
买方支付 2.5% + 卖方支付 2.5%
    ↓
两方都付款完成
    ↓
正式进入交易流程（备货→运输→完成）
    ↓
后续货款由买卖双方自行结算（线下打款 / 微信转账）
```

#### 3.7.3 异常处理
- 一方付款，另一方未付：等待 24 小时，超时自动取消
- 双方都未付款：撮合失效，返回需求池
- 交易取消：撮合费已收，**不予退还**（除非平台原因）
- 交易纠纷：代理人介入处理

### 5.2 SaaS 多租户原则
- **数据完全隔离**，租户之间不可跨平台撮合
- **App 名称和图标不可自定义**（保持品牌统一）
- **支付走各租户自己的账户**
- **每个租户有独立后台**

## 七、技术架构建议

### 7.1 前端
- **买家/卖家 App**：Flutter（一套代码，iOS + Android）
- **代理人后台**：Vue 3 + Element Plus
- **管理员后台**：Vue 3 + Element Plus
- **超级管理后台**：Vue 3 + Element Plus

### 7.2 后端
- **语言**：Node.js / Java / Go（推荐 Node.js + NestJS）
- **数据库**：MySQL / PostgreSQL（必须支持 `tenant_id` 字段，为 SaaS 做准备）
- **缓存**：Redis
- **消息队列**：RabbitMQ / Kafka

### 7.3 关键服务
- **AI 对话**：接入 DeepSeek / 通义千问 API
- **支付**：支付宝 SDK + 微信支付 SDK
- **短信**：阿里云短信服务
- **推送**：极光推送 / 友盟推送

### 7.4 多租户准备
**第一期所有数据库表都必须包含 `tenant_id` 字段**，即使现在只有山东一个租户。这样第二期开放 SaaS 时，不需要重构数据库。
