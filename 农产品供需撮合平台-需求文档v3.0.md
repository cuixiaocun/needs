## 一、产品概述

### 1.1 产品定位

> **用 AI 高效收集农产品 B2B 批发的供需信息，智能撮合菜农与采购方，减少中间环节，降低流通成本。**

### 1.2 业务范围
- **业务类型**：B2B 批发（不是 B2C 零售）
- **卖方**：菜农、果农、合作社
- **买方**：超市采购、批发商、餐饮企业
- **商品**：蔬菜、水果等农产品
- **地区**：山东先试点，可销售全国，后期 SaaS 复制

### 1.3 核心价值
- **效率提升**：AI 对话替代传统电话沟通
- **成本降低**：减少中间环节
- **智能匹配**：多维度自动撮合
- **服务托底**：代理人处理复杂场景
- **价格透明**：服务费明码标价
- **防跳单**：IM 中转 + 信用体系保护平台利益


## 五、模块详细设计

### 5.1 模块一：AI 聊天收集需求

#### 5.1.1 技术方案
- 接入第三方大模型 API（推荐 DeepSeek / 通义千问）
- 自定义 Prompt 模板，控制对话边界
- 结构化提取需求字段

#### 5.1.2 收集字段

| 角色 | 字段 |
|------|------|
| **卖方（菜农）** | 品类、数量、等级、产地、可供日期、期望价格 |
| **买方（采购方）** | 品类、数量、质量要求、到货时间、目标价格、配送地址 |

#### 5.1.3 注册流程（先聊后注册）

```
用户进入 App
    ↓
首屏弹窗：「同意《用户协议》和《隐私政策》」
    ↓
进入 AI 对话（对话内容存在设备本地）
    ↓
AI 提问 → 用户回答 → 生成需求卡片
    ↓
用户点击「退出」
    ↓
弹窗：「您的对话内容很有价值，是否保留？
       保留需要绑定手机号，下次进入可继续」
    ↓
  ┌─选「保留」→ 手机号注册 → 内容上传到服务器
  └─选「不保留」→ 直接退出，本地数据清空
```

#### 5.1.4 关键设计点
- **未注册阶段**：对话内容仅存在设备本地（合规要求）
- **注册后**：内容上传到服务器，关联到账户
- **对话保留期**：注册后永久保留，未注册的本地数据 7 天后自动清理

### 5.2 模块二：智能撮合

#### 5.2.1 撮合维度
- 品类匹配
- 数量匹配（±20% 区间）
- 地区匹配（同省优先 → 邻省次之 → 全国）
- 时间匹配（可供期 vs 到货期）
- 价格区间匹配（±10% 区间）

#### 5.2.2 撮合流程

```
需求入库
    ↓
多维度算法打分（每个维度 0-100 分）
    ↓
找到匹配度 Top 3 的对方
    ↓
推送通知给双方
    ↓
双方查看对方的需求卡片（不显示真实联系方式）
    ↓
双方点击「接受撮合」
    ↓
[此刻锁定当前费率] → 进入 IM 沟通阶段
```

### 5.4 模块四：站内 IM 沟通 ⭐ 新增防跳单核心

#### 5.4.1 功能定位
**所有买卖双方的沟通必须通过平台 IM 进行**，目的：
- 防止跳过平台直接交易
- 留存沟通记录用于争议处理
- 平台可监控异常行为

#### 5.4.2 核心规则
- **不显示对方真实手机号**
- **不显示对方真实姓名**（只显示昵称）
- **禁止发送外部联系方式**（系统识别+屏蔽手机号、微信号）
- **支持图片、视频**（让买家看货）
- **聊天记录留存 90 天**（用于争议处理）

#### 5.4.3 防跳单识别
- 发送疑似手机号 → 自动屏蔽 + 警告
- 发送疑似微信号 → 自动屏蔽 + 警告
- 多次违规 → 信用扣分 → 严重者封号

### 5.6 模块六：撮合服务费收取 ⭐ 重点重构

#### 5.6.1 收费规则

| 项目 | 规则 |
|------|------|
| **总费率** | **1%**（默认值，可后台调整） |
| **卖方费率** | **0.7%**（默认，可单独配置） |
| **买方费率** | **0.3%**（默认，可单独配置） |
| **收费时机** | **成交后收费**（买方确认收货） |
| **首单优惠** | **每个用户第一单免费** |
| **支付方式** | 支付宝 / 微信支付 |
| **资金流向** | 直接进入平台账户 |
| **费率锁定** | **双方接受撮合时锁定**，进行中订单不受变更影响 |

#### 5.6.2 收费流程图

```
订单成交（买方确认收货）
    ↓
系统检查：是不是用户首单？
    ↓
  ┌─是首单 → 免费 → 完成
  └─不是首单 → 进入下面流程
    ↓
按订单成交时锁定的费率计算服务费
  - 卖方应付 = 订单金额 × 卖方费率
  - 买方应付 = 订单金额 × 买方费率
    ↓
推送付款通知给买卖双方
    ↓
双方在 7 天内完成支付
    ↓
[如有代理人介入] 自动计算代理人提成（20%）
    ↓
平台净收入 = 服务费 - 代理人提成
```

#### 5.6.3 计费示例

**示例 1：常规订单**
```
订单金额：10,000 元
卖方费率：0.7% → 卖方付 70 元
买方费率：0.3% → 买方付 30 元
总服务费：100 元
（无代理人介入）
平台净收入：100 元
```

**示例 2：代理人介入订单**
```
订单金额：10,000 元
卖方费率：0.7% → 卖方付 70 元
买方费率：0.3% → 买方付 30 元
总服务费：100 元
代理人提成：100 × 20% = 20 元
平台净收入：100 - 20 = 80 元
```

**示例 3：用户首单**
```
订单金额：10,000 元
首单免费 → 双方都不付服务费
平台净收入：0 元
（但获得了用户和数据）
```

```
├────────────────────────────────────────┤
│  待生效费率                            │
│  ────────────────────                  │
│  卖方费率：1.0%                        │

└────────────────────────────────────────┘
```

#### 5.7.3 费率变更规则

| 规则 | 说明 |
|------|------|
| **生效日期** | 必须填写未来日期，不能立即生效 |
| **进行中订单** | 已经「双方接受撮合」的订单按旧费率，不受影响 |
| **新订单** | 在生效日期当天 0 点后的新订单使用新费率 |
| **历史记录** | 所有费率变更必须留存历史记录，便于审计 |
| **首单标记** | 用户首单免费的标记独立于费率，不受影响 |

### 5.8 模块八：代理人系统 + 提成机制 ⭐ 重点重构

#### 5.8.1 代理人定位
- 平台自有团队
- 第一期 2 人
- **靠提成激励，无需排班**
- 处理 AI 无法解决的复杂场景

#### 5.8.2 提成规则（重点）

| 订单类型 | 平台提成 | 代理人提成 |
|---------|---------|----------|
| **AI 自动撮合** | 100% | **0%** |
| **代理人介入撮合** | 80% | **20%** |
| **代理人接单失败** | - | 0%（但记录工作量） |

**举例**：
```
订单金额 10,000 元，总服务费 100 元

情况 A：AI 自动撮合
  → 平台净收入 100 元
  → 代理人提成 0 元

情况 B：代理人介入
  → 平台净收入 80 元
  → 代理人提成 20 元
```

#### 5.8.3 代理人在线状态机制

```
代理人后台
    ↓
顶部状态栏：[在线接单] / [离线]
    ↓
选择「在线接单」时才会被自动分配任务
    ↓
选择「离线」时不会接到新任务
```

#### 5.8.4 任务分配逻辑

```
用户点击「转人工」
    ↓
系统查找在线代理人
    ↓
  ┌─有在线代理人 → 按当前任务量轮询分配（任务最少的优先）
  └─无在线代理人 → 提示用户：「人工服务时间 9:00-21:00，
                              当前所有客服离线，您的需求将被记录，
                              客服上线后会主动联系您」
    ↓
代理人接单 → 处理 → 完成
```

#### 5.8.5 代理人后台功能

| 功能模块 | 包含内容 |
|---------|---------|

| **提现申请** | 提现金额、记录、审核状态 |

```
代理人联系双方了解情况
    ↓
代理人作出裁决
    ↓

    ↓
```

## 六、SaaS 多租户架构（第二期）

### 6.1 山东版本的双重身份明确化

**重要：明确分离两个角色**

```
角色 1：你 = 平台所有者（超级管理员）
        ↓
        管理整个 SaaS 平台

角色 2：山东运营公司 = 第一个租户
        ↓
        是 SaaS 系统中的一个普通租户
        但你是这个公司的法人

财务关系：
  - 平台公司不向山东租户收 SaaS 费（自己人）
  - 但数据上山东就是普通租户，走标准流程
  - 山东的服务费收入归"山东运营公司"
  - 平台收入归"平台公司"
```

### 6.2 多租户架构图

```
┌─────────────────────────────────┐
│      超级管理后台（你专用）       │
│   - 管理所有租户                  │

└──────────────┬──────────────────┘
               │
       ┌───────▼────────┐
       │  SaaS 平台核心  │
       └───┬───┬────┬───┘
           │   │    │
   ┌───────▼───▼────▼───────────┐
   │      多租户隔离层            │
   └───┬───┬───┬───┬─────────────┘
       │   │   │   │
   ┌───▼┐ ┌▼┐ ┌▼┐ ┌▼──────────────┐
   │租户A│ │B │ │C │ │ 山东运营公司   │
   └─┬──┘ └┬─┘ └┬─┘ └────┬──────────┘
     │     │    │        │
   ┌─▼─────▼────▼────────▼──┐
   │  各自的 App 端          │
   │  （买家/卖家 App）       │
   │  （代理人/管理 Web 后台）│
   └────────────────────────┘
```

### 6.3 SaaS 多租户原则
- **数据完全隔离**，租户之间不可跨平台撮合
- **App 名称和图标不可自定义**（保持品牌统一）
- **支付走各租户自己的账户**
- **每个租户有独立后台**

## 七、技术架构建议

### 7.1 前端
- **买家/卖家 App**：Flutter（一套代码，iOS + Android）
- **代理人后台**：Vue 3 + Element Plus（Web）
- **管理员后台**：Vue 3 + Element Plus（Web）
- **超级管理后台**：Vue 3 + Element Plus（Web）

### 7.2 后端
- **语言**：Node.js + NestJS（推荐）/ Java + Spring Boot
- **数据库**：MySQL / PostgreSQL
- **缓存**：Redis
- **消息队列**：RabbitMQ / Kafka
- **IM 服务**：自建（WebSocket）/ 接入环信、融云

### 7.3 关键服务
- **AI 对话**：DeepSeek / 通义千问 API
- **支付**：支付宝 SDK + 微信支付 SDK
- **短信**：阿里云短信服务
- **推送**：极光推送 / 友盟推送
- **运输 API**：满帮、货拉拉、快狗打车

### 7.4 多租户准备（关键！）
**第一期所有数据库表都必须包含 `tenant_id` 字段**：
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,  -- 关键字段
    phone VARCHAR(20),
    ...
    INDEX idx_tenant (tenant_id)
);
```
即使第一期只有山东一个租户（tenant_id = 1），也要预留这个字段。这样第二期开放 SaaS 时，不需要重构数据库。

### 7.5 防跳单技术实现
- **IM 中转**：所有沟通必经过平台 WebSocket
- **手机号识别**：正则表达式 + 关键词屏蔽
- **微信号识别**：正则表达式 + 关键词屏蔽
- **图片 OCR**（可选）：识别图片中的联系方式
- **行为分析**：异常聊天模式识别

## 九、关键流程图汇总

### 9.1 完整成交流程（含费率锁定）

```
撮合成功
    ↓
通知双方
    ↓
双方点击「接受」← ⭐ 此刻锁定费率（费率版本号记录在订单中）
    ↓
进入 IM 沟通（防跳单）
    ↓
卖方备货 → 接入运输 API → 运输中
    ↓
买方确认收货
    ↓
[关键节点] 系统检查：是用户首单吗？
    ↓
  ┌─是首单 → 跳过付款 → 完成
  └─非首单 → 按锁定费率计算 → 双方付款 → 完成
    ↓
[如有代理人介入] 自动计算代理人提成
```

### 9.2 防跳单完整机制

```
用户在 IM 中尝试发送手机号
    ↓
系统正则识别 → 拦截 + 警告
    ↓
[如果继续] → 信用扣分
    ↓
[严重违规] → 封号

代理人定期回访
    ↓
发现用户多次撮合后没成交但仍活跃
    ↓
重点关注，调查是否跳单
    ↓
确认跳单 → 信用扣分 + 警告
```

### 9.3 费率变更流程

```
管理员进入费率管理后台
    ↓
新增费率变更：
  - 卖方费率（新）
  - 买方费率（新）
  - 生效日期（必须未来日期）
    ↓
保存 → 进入「待生效」状态
    ↓
到达生效日期
    ↓
自动启用新费率
    ↓
新订单使用新费率
进行中订单（已接受撮合的）继续用旧费率
```

## 十、财务模型示例

### 10.1 月度收入预估（理论值）

**假设场景**：
- 山东试点第 6 个月
- 月活用户 1000 人
- 月成交订单 200 单
- 平均订单金额 5000 元

**收入计算**：
```
非首单订单：200 × 80% = 160 单（20% 是首单免费）
平均服务费 = 5000 × 1% = 50 元
月服务费收入 = 160 × 50 = 8000 元

代理人介入比例：30%
代理人介入订单 = 160 × 30% = 48 单
代理人提成 = 48 × 50 × 20% = 480 元

平台净收入 = 8000 - 480 = 7520 元
```

### 10.2 收入优化路径

```
阶段 1（第 1-3 月）：免费期
  - 服务费 = 0
  - 目标：冲规模，验证产品

阶段 2（第 4-6 月）：低费率
  - 服务费 = 0.5%
  - 目标：开始变现，培养付费习惯

阶段 3（第 7-12 月）：标准费率
  - 服务费 = 1%
  - 目标：稳定收入

阶段 4（第 13 月+）：差异化定价
  - 大单 0.5%，小单 1.5%
  - 目标：优化利润
```
