## 一、产品概述

### 1.1 产品定位

> 用 AI 高效收集农产品供需信息，智能撮合买卖双方与运输方，减少中间环节，降低流通成本。

### 1.2 核心价值

- **效率提升**：通过 AI 对话替代传统的人工沟通，快速收集供需信息
- **成本降低**：减少中间环节，让农产品流通更直接
- **智能匹配**：根据多维度自动撮合买卖双方与运输方
- **服务托底**：代理人系统处理 AI 无法解决的复杂场景

### 1.3 业务场景

聚焦农产品（蔬菜、水果等）流通领域，连接：
- **生产端**：果农、菜农、合作社
- **采购端**：超市、批发商、餐饮企业
- **物流端**：物流公司、个人司机

## 二、用户角色

### 2.1 角色清单

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| **卖方** | 果农、菜农、合作社 | 快速找到买家，卖个好价钱 |
| **买方** | 超市、批发商、餐饮企业 | 稳定供货，质量可靠，价格透明 |
| **运输方** | 物流公司、个人司机 | 找到合适的货单，减少空车 |
| **代理人** | 平台客服 / 经纪人（自有团队） | 处理复杂谈判、纠纷、特殊需求 |
| **管理员** | 平台运营人员 | 监控数据、审核用户、管理内容 |

### 2.2 角色关系图

```
   卖方 ──┐                    ┌── 买方
          │                    │
          └──→ 平台撮合 ←──────┘
                  ↓
              运输方
                  ↑
            代理人介入（必要时）
                  ↑
              管理员监控
```

### 3.2 模块一：AI 聊天收集需求

**核心原则**：简洁无废话，像填表一样高效。

#### 3.2.1 卖方收集字段
- 品类
- 数量
- 等级
- 产地
- 可供日期
- 期望价格

#### 3.2.2 买方收集字段
- 品类
- 数量
- 质量要求
- 到货时间
- 目标价格
- 配送地址

#### 3.2.3 运输方收集字段
- 车型
- 载重
- 区域
- 可出发时间
- 报价

#### 3.2.4 工作流程

```
用户进入 → AI 提问 → 用户回答 → 生成结构化需求卡片 → 用户确认 → 发布
```

## 五、商业模式

### 5.1 收费方式总览

| 收费方式 | 对象 | 金额参考 | 说明 |
|---------|------|---------|------|
| **成交佣金** | 买卖双方 | 0.5% - 2% | 核心收入，撮合成功才收 |

| **代理人服务费** | 复杂订单 | 按单收费 | 人工介入额外收费 |
| **SaaS 授权费** | 第三方客户 | 见 5.2 | 平台多租户授权 |

## 六、平台架构

### 6.1 多租户架构示意

```
┌─────────────────────────────────┐
│      超级管理后台（自有）        │
└──────────────┬──────────────────┘
               │
       ┌───────▼────────┐
       │  SaaS 平台核心  │
       └───┬───┬────┬───┘
           │   │    │
       ┌───▼┐ ┌▼─┐ ┌▼──────────┐
       │租户│ │租│ │ 山东本部   │
       │ A  │ │B │ │（首个租户）│
       └─┬──┘ └┬─┘ └────┬──────┘
         │     │        │
       ┌─▼─────▼────────▼──┐
       │  各自 App 端       │
       │ (买/卖/代理人)     │
       └───────────────────┘
```

### 6.2 租户后台功能
- 用户管理
- 需求审核
- 代理人管理
- 财务结算
- 品类配置
- 佣金设置
- 数据报表

### 6.3 超级管理后台功能
- 所有租户管理
- 全平台数据汇总
- 租户费用管理
- 系统配置

### 6.4 租户隔离原则
- 数据**完全隔离**，不可跨平台撮合
- App 名称和图标**不可自定义**（保持品牌一致）
- 支付走各租户自己的账户

## 九、关键业务决策

### 9.1 地理范围策略
- **第一阶段**：以山东为试点
- **第二阶段**：通过 SaaS 模式复制到其他地区
- 卖家可销售到全国，不限于山东

### 9.2 平台架构策略
- 自营山东版本作为第一个租户测试
- 山东版本跑通后，才向外界销售 SaaS

### 9.3 代理人模式
- 完全由**自有团队**组成
- 不招募第三方经纪人
- 专门处理 AI 无法处理的复杂问题

### 9.4 优先级顺序
1. 第一优先：验证山东市场，跑通 MVP
2. 第二优先：完善支付和评价体系
3. 第三优先：开放多租户，开始售卖 SaaS
