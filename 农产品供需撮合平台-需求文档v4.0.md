## 📋 v4.0 重大变更说明

### 与 v3.0 的根本性差异

| 维度 | v3.0 | v4.0 |
|------|------|------|
| **撮合模式** | 双方互相联系 | **暗池撮合**（双方互相不知道对方） |
| **沟通方式** | 站内 IM | **无 IM**（只和 AI 对话） |
| **资金流转** | 双方私下结算 | **支付宝担保交易**（平台中转） |
| **物流安排** | 卖家自己叫车 | **平台统一调度 + 自动拼车** |
| **质量保障** | 双方自行确认 | **标准等级 + 图片 + 退货** |
| **核心竞争力** | 撮合效率 | **AI 拆单/拼单/拼车 + 暗池撮合** |
| **跳单问题** | 难以根治 | **从根本上消除**（双方不联系） |
| **信任问题** | 依赖信用积分 | **平台资金托管 + 标准化等级** |

### v4.0 的核心理念

> **类比股票交易所**：
> 卖家挂单（卖单）+ 买家挂单（买单）→ AI 系统自动撮合 → 完成交易
> 双方完全互不联系，平台是唯一的中间人

## 一、产品概述

### 1.1 产品定位

> **农产品智能撮合交易所**：用 AI 撮合菜农和采购方，双方完全不联系，平台全程托管，实现农产品 B2B 批发的暗池交易。

### 1.2 与传统模式的对比

| 维度 | 传统农批市场 | 一般撮合平台 | **本平台** |
|------|------------|------------|----------|
| **买卖双方接触** | 直接面对面 | 平台介绍后联系 | **完全不联系** |
| **价格谈判** | 当面议价 | 双方私聊 | **AI 自动撮合区间** |
| **货款流转** | 现金/转账 | 双方自行 | **平台担保托管** |
| **质量保障** | 看货决定 | 自行判断 | **标准等级 + 退货** |
| **物流** | 自己安排 | 自己叫车 | **平台统一调度 + 拼车** |

### 1.3 核心价值

- **零跳单风险**：双方不联系，无法绕过平台
- **零信任成本**：资金担保 + 标准等级 + 退货保障
- **极致效率**：AI 自动撮合 + 拼单 + 拼车
- **价格透明**：明码标价，无中间盘剥
- **物流优化**：智能拼车，降低成本 30-60%

### 1.4 业务范围

- **业务类型**：B2B 批发（不是 B2C 零售）
- **卖方**：菜农、果农、合作社
- **买方**：超市采购、批发商、餐饮企业
- **商品**：蔬菜、水果等农产品
- **地区**：山东先试点，可销售全国，后期 SaaS 复制

### 5.3 模块二：撮合引擎（核心）

#### 5.3.1 撮合优先级

```
撮合引擎每 5 分钟扫描一次

优先级 1：完全匹配（数量、等级、价格、地点都精确）
    ↓ 找不到
优先级 2：单卖家拆单（卖家有大货 → 拆给多买家）
    ↓ 找不到
优先级 3：（第二期）多卖家拼单
    ↓ 找不到
优先级 4：进入等待池，新订单进来时重新撮合
    ↓ 7 天后仍未撮合
优先级 5：通知用户「未匹配到合适订单」+ 提供调整建议
```

#### 5.3.2 价格匹配算法

```
定义：
  卖家价格区间：[卖家最低价, 卖家期望价] = [Sl, Sh]
  买家价格区间：[买家目标价, 买家最高价] = [Bl, Bh]

匹配条件：两个区间必须有交集
  即 max(Sl, Bl) ≤ min(Sh, Bh)

成交价格：取交集的中间值
  成交价 = (max(Sl, Bl) + min(Sh, Bh)) / 2

举例 1（典型）：
  卖家 [3.3, 3.5]，买家 [3.4, 3.6]
  交集 [3.4, 3.5]
  成交价 = (3.4 + 3.5) / 2 = 3.45 元

举例 2（卖家便宜）：
  卖家 [3.0, 3.2]，买家 [3.4, 3.6]
  交集为空（3.2 < 3.4）→ 不匹配
  但买家更贵 → AI 建议：「找到更便宜的卖家，是否接受 3.2 元/斤？」

举例 3（买家便宜）：
  卖家 [3.5, 3.8]，买家 [3.0, 3.3]
  交集为空 → 不匹配
  AI 建议买家提高出价，或建议卖家降低底价
```

#### 5.3.3 数量匹配算法

```
完全匹配：数量相等
拆单匹配：卖家数量 ≥ 多个买家数量之和
等待匹配：数量不等且无法拆分

举例：
  卖家 A：5000 斤花生
  买家 X：2000 斤
  买家 Y：1500 斤
  买家 Z：1500 斤

  AI 拆单：
  - 订单 1：A → X，2000 斤
  - 订单 2：A → Y，1500 斤
  - 订单 3：A → Z，1500 斤
  - 卖家 A 一次性出货 5000 斤，分送 3 个地点
```

### 5.4 模块三：智能拆单 ⭐ 核心竞争力

#### 5.4.1 拆单触发条件

```
卖家 5000 斤花生
    ↓
撮合引擎扫描买单池
    ↓
没有 5000 斤的买家
    ↓
但有多个小数量买家
    ↓
触发拆单逻辑
```

#### 5.4.2 拆单算法

```
1. 按价格排序买家（出价高的优先）
2. 累计买家数量，直到接近卖家总数量
3. 检查：
   - 配送地点是否合理（同区域优先）
   - 时间是否合理（卖家可供期 vs 买家到货期）
4. 生成多个子订单
5. 通知卖家：「您的 5000 斤已售出，分给 3 个买家」
6. 通知 3 个买家：「您的订单已匹配」
```

#### 5.4.3 拆单后的物流

```
方案 A：卖家送到一个集散点 → 平台分拨
  - 卖家：开车送到集散点（一次）
  - 平台：分拨给 3 个买家（拼车）

方案 B：卖家直接分送
  - 一辆车跑 3 个地方
  - 适合同区域的买家

方案 C：物流公司一次取货 + 多点配送
  - 物流车到卖家处取货
  - 沿途配送给多个买家
```

**第一期推荐方案 C**：交给运输公司处理。

### 5.5 模块四：智能拼单（第二期）

#### 5.5.1 拼单触发条件

```
买家 X 要 5000 斤花生
    ↓
撮合引擎扫描卖单池
    ↓
没有单个卖家有 5000 斤
    ↓
但有多个小数量卖家
    ↓
触发拼单逻辑
```

#### 5.5.2 拼单逻辑

```
1. 按价格排序卖家（价格低的优先）
2. 累计卖家数量，直到达到买家需求
3. 检查：
   - 卖家产地是否相近（同区域优先）
   - 等级是否一致（避免参差不齐）
4. AI 询问买家：「找到 3 个卖家可拼单提供 5000 斤
                平均价格 3.4 元，是否接受？」
5. 买家同意 → 生成多个子订单
```

**第一期暂缓**：拼单涉及多卖家协调，复杂度高，第二期再做。

### 5.6 模块五：智能拼车 ⭐ 核心竞争力

#### 5.6.1 拼车逻辑

```
平台每小时扫描所有订单
    ↓
找到同方向、时间相近的订单
    ↓
计算拼车方案：
  - 一辆 5 吨车
  - 起点：A 区域
  - 终点：B 区域
  - 装载多个订单的货物
    ↓
对比单独运输 vs 拼车成本
    ↓
拼车成本更低 → 自动拼车
通知所有相关订单：「您的订单将与其他订单拼车配送」
```

#### 5.6.2 拼车示例

```
情况 1：单独运输
  - 订单 A：临沂 → 寿光，运费 800 元
  - 订单 B：临沂 → 寿光，运费 800 元
  - 订单 C：临沂 → 寿光，运费 800 元
  - 总运费：2400 元

情况 2：拼车
  - 三个订单合并到一辆车
  - 拼车运费：1200 元
  - 节省：1200 元（50%）
  - 每个订单分摊：400 元

节省比例：50%
```

#### 5.6.3 拼车规则

| 规则 | 说明 |
|------|------|
| 自动拼车 | 默认开启，用户可选择「我要单独运输」 |
| 拼车优惠 | 拼车成功的订单，运费按比例减免 |
| 拼车失败 | 凑不齐拼车 → 按原运费单独运输 |
| 等待时间 | 最长等待 24 小时拼车，超时单独发 |

### 5.7 模块六：标准化等级体系 ⭐

#### 5.7.1 等级标准

每种农产品都有标准化等级，挂单时**强制选择**：

**示例：花生等级**
```
特级：颗粒饱满，无破损，无霉变，含水量 < 8%，每 100g ≥ 200 颗
一级：颗粒较饱满，破损率 < 5%，无霉变，含水量 < 9%
二级：颗粒一般，破损率 < 15%，少量瑕疵，含水量 < 10%
三级：可接受少量损耗，破损率 < 25%
```

**示例：白菜等级**
```
特级：菜心饱满，外叶完整，无虫害，重量 > 3 斤/颗
一级：菜心较饱满，少量外叶损伤，重量 2-3 斤/颗
二级：可接受瑕疵，重量 1-2 斤/颗
```

#### 5.7.2 等级与价格

平台为每个等级提供**参考价格区间**，引导用户合理定价：
```
花生（一级）参考价：
  - 山东产地：3.0 - 3.6 元/斤
  - 历史成交均价：3.35 元/斤
  - 当日大宗市场价：3.40 元/斤
```

#### 5.7.3 等级争议处理

```
买家收到货 → 觉得等级不符
    ↓
申请「等级争议」→ 上传图片证据
    ↓
代理人介入 → 查看双方图片
    ↓
代理人判定：
  - 卖家虚标等级 → 退款 + 信用扣分
  - 买家无理取闹 → 不退款 + 买家信用扣分
  - 等级有争议 → 部分退款（如 30%）
```

### 5.8 模块七：支付宝担保交易 ⭐

#### 5.8.1 资金流转图

```
匹配成功 → 双方确认订单
    ↓
[关键节点 1] 买家在 App 内付款
    付款金额 = 货款 + 服务费 + 运输费
    付款方式 = 支付宝
    ↓
钱进入支付宝担保账户（不进入卖家账户）
    ↓
通知卖家：「订单已付款，请发货」
    ↓
卖家发货 → 物流跟踪
    ↓
运输方标记「已送达」
    ↓
7 天异议期
    ↓
[关键节点 2] 无异议 / 异议处理完毕
    ↓
平台触发支付宝分账：
  - 货款 → 卖家支付宝账户
  - 服务费 → 平台支付宝账户
  - 运输费 → 运输方支付宝账户
    ↓
订单完成
```

#### 5.8.2 支付宝担保交易的优势

- ✅ 不需要平台自己申请支付牌照
- ✅ 不需要建立资金池
- ✅ 用户对支付宝信任度高
- ✅ 交易流程标准化
- ✅ 退款流程自动化

#### 5.8.3 接入方案

- 接入支付宝「**担保交易**」接口
- 或接入「**支付宝商家服务**」
- 文档参考：alipay.com/open-platform

### 5.9 模块八：物流统一调度

#### 5.9.1 物流流程

```
订单付款成功
    ↓
平台调度系统
    ↓
分析订单：
  - 取货地点：卖家产地
  - 送货地点：买家地址
  - 数量、重量
  - 是否可拼车
    ↓
方案 1：拼车（同方向多订单）
    ↓
方案 2：单独发车
    ↓
调用第三方运输 API（满帮 / 货拉拉 / 快狗打车）
    ↓
获取运输方报价
    ↓
选择最优方案 → 下单
    ↓
运输方接单 → 取货 → 运输 → 送达
    ↓
更新订单状态
```

#### 5.9.2 第三方运输 API 接入

| 平台 | 适用场景 | 接入难度 |
|------|---------|---------|

| **货拉拉** | 同城短途 | 中（开放 API） |
| **快狗打车** | 城市配送 | 中 |

**第一期建议**：先接入货拉拉（最易接入），覆盖同城和短途。

#### 5.9.3 运输费计算

```
运输费 = 基础运费 + 公里费 × 距离 + 重量附加费
      ↓
   平台显示给买家：
   「运输费：300 元」（明码标价）
      ↓
   买家付款时一起付
```

### 5.10 模块九：退货与仲裁机制 ⭐

#### 5.10.1 异议期机制

```
运输方标记「已送达」 → 进入 7 天异议期
    ↓
买家有 7 天时间检查货物
    ↓
  ┌─无异议 → 7 天后自动确认 → 平台分账
  └─有异议 → 申请退货 / 退款
            ↓
            代理人介入
            ↓
            查看双方证据
            ↓
            裁决：
              - 全额退款
              - 部分退款
              - 不退款
            ↓
            执行结果
```

#### 5.10.2 退款规则

| 场景 | 处理方式 |
|------|---------|
| 货物未送达 | 100% 退款（货款 + 运费） |
| 货物完全损坏 | 100% 退款 + 卖家信用扣分 |
| 等级严重不符 | 100% 退款 + 卖家信用扣分 |
| 等级轻微不符 | 部分退款（30-50%） |
| 数量少于约定 | 按差额退款 |
| 买家无理由拒收 | 不退款 + 买家信用扣分 |
| 平台原因撤单 | 100% 退款 |
| 双方协商取消 | 退款（扣 5% 服务成本） |

#### 5.10.3 代理人仲裁流程

```
争议产生
    ↓
系统自动通知所有在线代理人
    ↓
按轮询规则分配给代理人 A
    ↓
代理人 A 查看：
  - 订单详情
  - 卖家挂单时上传的图片
  - 买家收货后上传的图片
  - 物流签收记录
  - 历史交易记录
    ↓
代理人 A 联系双方了解情况（通过 AI 中转）
    ↓
代理人 A 作出裁决
    ↓
裁决结果触发支付宝分账（部分退款 / 全额退款）
    ↓
代理人获得裁决补贴（每单 10 元）
    ↓
留存仲裁记录
```

## 七、技术架构建议

### 7.1 前端
- **买家/卖家 App**：Flutter（一套代码，iOS + Android）
- **代理人后台**：Vue 3 + Element Plus
- **管理员后台**：Vue 3 + Element Plus
- **超级管理后台**：Vue 3 + Element Plus

### 7.2 后端
- **语言**：Node.js + NestJS（推荐）
- **数据库**：MySQL / PostgreSQL（必须支持 `tenant_id`）
- **缓存**：Redis
- **消息队列**：RabbitMQ / Kafka
- **撮合引擎**：独立服务，定时扫描挂单池

### 7.3 关键服务
- **AI 对话**：DeepSeek API（成本低）
- **支付**：支付宝担保交易 SDK
- **实名认证**：阿里云实名认证 API
- **运输 API**：货拉拉 API
- **短信**：阿里云短信
- **推送**：极光推送

### 7.4 撮合引擎技术方案

```
撮合引擎独立服务
    ↓
定时任务：每 5 分钟扫描一次挂单池
    ↓
扫描所有「待撮合」状态的订单
    ↓
对每个卖单：
  - 在买单池中查找匹配的买单
  - 按优先级（精确匹配 → 拆单 → 拼单）
  - 找到匹配 → 创建订单 → 通知双方
  - 未找到 → 标记为「等待中」
    ↓
对每个买单：同上
    ↓
更新数据库
```

### 7.5 多租户准备
所有数据库表都必须包含 `tenant_id` 字段：
```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    seller_id BIGINT,
    buyer_id BIGINT,
    ...
    INDEX idx_tenant (tenant_id)
);
```

## 九、财务模型示例

### 9.1 单笔订单收入

```
订单：花生 2000 斤 × 3.45 元 = 6900 元

买家付款：
  - 货款：6900 元
  - 服务费：6900 × 1% = 69 元
  - 运输费：300 元
  - 总计：7269 元
    ↓
进入支付宝担保账户
    ↓
确认收货后分账：
  - 卖家：6900 元（货款）
  - 平台：69 元（服务费）
  - 运输方：300 元（运输费）
```

### 9.2 月度收入预估（理论值）

**假设场景**：
- 山东试点第 6 个月
- 月成交订单 200 单
- 平均订单金额 5000 元
- 首单免费比例 20%

**收入计算**：
```
非首单订单：200 × 80% = 160 单
平均服务费 = 5000 × 1% = 50 元
月服务费收入 = 160 × 50 = 8000 元

代理人提成（30% 介入）：
  48 单 × 50 × 20% = 480 元

代理人仲裁补贴（10% 争议）：
  20 单 × 10 = 200 元

平台净收入 = 8000 - 480 - 200 = 7320 元/月
```

### 9.3 拼车带来的物流节省

```
不拼车情况：
  200 单 × 平均运费 500 元 = 100,000 元

拼车情况（50% 拼车率，节省 40%）：
  100 单 × 500 + 100 单 × 300 = 80,000 元

为用户节省：20,000 元/月
（这是平台核心竞争力之一）
```
